# conversation-store

## MODIFIED Requirements

### Requirement: Flow events bridge widget completions into agent memory
When a deterministic report endpoint resolves — a download or email success, OR a no-data / failed outcome — the backend SHALL append a `flow_event` turn to the session's thread through the existing store path (synchronous enqueue; the write MUST NOT be awaited by, delay, or ever fail the report response — store absent or degraded means the event is skipped silently). The turn SHALL be `role: "user"`, `kind: "flow_event"`, with `content` holding one deterministic rendered memo text block framed as an app event — for a success, "[App event — generated by the app, not typed by the user] The user completed the <flow> form in the widget: <slot labels>, <delivery outcome>."; for a no-data / failed attempt, an equivalently framed memo stating the requested <flow> and <slot labels> and that no records were found (or the attempt did not complete) — and `meta` carrying the structured fields (flow, slots, delivery, outcome), where `outcome` distinguishes success from the no-data / failed case. Thread replay (`messages()`) SHALL include `flow_event` turns as user-role content, merging into an adjacent user message where the wire would otherwise produce consecutive user roles; the stored memo text SHALL be byte-identical to what the model receives. Memos SHALL contain only whitelisted slot labels and ISO dates — never credentials, file tokens, or upstream data.

#### Scenario: Widget completion becomes context
- **WHEN** the user completes a seeded P&L form and the report endpoint succeeds
- **THEN** a `flow_event` turn lands in the thread and the next `/api/chat` call replays it to the model as the app-event memo

#### Scenario: No-data attempt becomes context
- **WHEN** the user completes a P&L form and the endpoint returns no data for that period
- **THEN** a `flow_event` turn with `outcome` "no_data" lands in the thread, and the next `/api/chat` call replays it so the model knows the attempt was made and returned nothing

#### Scenario: Report response never blocked
- **WHEN** the store queue is full or the database is down while a report generates
- **THEN** the report response is unaffected and the flow event is dropped with only a counter/log record

#### Scenario: Memo is data, not instruction
- **WHEN** a flow_event memo is replayed
- **THEN** it is framed as an app event distinct from user prose, and contains only the flow name, slot labels, delivery mode, and outcome
