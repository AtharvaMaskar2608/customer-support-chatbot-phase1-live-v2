00 Orientation

Two things to hold in your head first

Android is a capture source, not an API surface

These were captured from the Android app, but the from: header is a client-build tag — not auth and not a source router (live-tested). The same session+client returned 200 with the web tag, the android tag, a made-up tag, and no from: header at all. One surface serves both web and android — never build source-specific endpoints. What varies per platform is the SessionId (minted per login), never the endpoint. The one open question is CML/MIS, whose JWT carries source: FINX_ANDROID — source-gating there is unconfirmed.

✓ Email is supported alongside download

The three .NET file endpoints (GetGlobalPNLPDF, GetLedgerDetailsPDF, GetTaxReportPDF) take a RequestFor field that selects download vs email. RequestFor: 1 = email is the only value constant across all three. Email is live-confirmed on P&L and Tax, and presumed on Ledger (not yet fired). CML and Contract Notes have no email mode.

01 Infrastructure

Three backends, three ways to authenticate

The error model has two layers

  • Business failures are in-band. HTTP 200 — branch on Status/StatusCode in the body. Response is null (not []) when there's no data.
  • Auth failures are a real HTTP 401. Detect by status code, not body shape — the two 401 envelopes differ: .NET sends {"Status":"Fail","Response":"","Reason":"Invalid SessionId"} with header authstatus: Unauthorized; MIS sends {"statusCode":401,"message":"Invalid Request."...}.
  • "Invalid SessionId" = session↔client mismatch, not necessarily an expired token. The .NET backend binds a SessionId to one ClientId; a valid session for X008593 returned 401 for a different client and 200 for its own. Always send the ClientId the session belongs to.

Rule

RequestFor = 1 = Email is the only cross-endpoint constant. The download value forks — Tax uses 2, P&L and Ledger use 0. Do not centralize RequestFor behind one shared constant — hardcode it per endpoint. FileFormat (1=PDF, 2=Excel) is consistent wherever it appears (Tax only, so far).

Generates the Client Master List. This is the JWT-authed backend — it takes the SSO accessToken, not the SessionId. The request shape reads like a generic report generator; only cml is confirmed.

Headers

http

source: FINX_ANDROID
authorization: <SSO JWT = frontend accessToken>   # NOT the SessionId
from: android_9.2.2.260710
content-type: application/json

Request — model CmlRequest

json
{
  "reportType": "cml",
  "searchBy": "client-id",
  "searchValue": "X04883"
}

Response — 200 · live-captured · camelCase envelope

json → CmlBody.cmlLink
{
  "statusCode": 200,
  "message": "URL generated successfully",
  "devMessage": null,
  "body": {
    "cmlLink": "https://onmedia.choiceindia.com/JF/<hash>_CML_<ts>.pdf?X-Amz-Expires=120&X-Amz-Signature=<MASKED>"
  }
}
  • Link field is body.cmlLink — an AWS S3 SigV4 pre-signed URL fronted by CloudFront.
  • Downloads as Client_Master_List.pdf (%PDF-1.6, 9002 bytes).
  • Object timestamp shows a static pre-generated S3 file — the PDF may be stale vs the current master list.
🔴 Security — Flag B

The X-Amz-Expires=120 "single-use" assumption is false in practice. CloudFront caches by path only (ignoring the signed query string), so once any valid link warms the edge the same S3 path is served for the CloudFront TTL regardless of signature expiry, and repeat fetches succeed. Don't treat 120s/single-use as a boundary — fetch server-side and never surface the link. Full detail →

Email specifics

On the three .NET endpoints, email mode returns Status: "Success" with Response = a confirmation string containing the full registered email, uppercased — always mask. Response is polymorphic; the FileDeliveryResponse model exposes is_download_url() / is_email_confirmation() to branch safely.

05 Live-production observations

Security flags

These are observations on live Choice/FinX infrastructure, not chatbot bugs — reported so the Jini backend can defend against them and the FinX team can be notified. Not exploited: a single X99999 probe confirmed Flag A; no real client IDs were enumerated.

FLAG A

Contract-note chain has no authentication (IDOR / BOLA)

Both /middleware-go/report/contract (list) and /contract/download authorized purely on the body client_id/client_code — returning HTTP 200 + full data with garbage or absent authorization. Any known client's contract notes and their file_id PDF handles appear retrievable unauthenticated.

Mitigation: enforce client-ownership server-side against the authenticated widget session; treat file_id as sensitive; report upstream.
FLAG B

CML signed-URL controls nullified by CloudFront caching

The cmlLink S3 SigV4 URL carries X-Amz-Expires=120, but CloudFront's cache key is path-only (ignores the signed query string). Once any valid link warms the edge cache, the same S3 path is served for the CloudFront TTL regardless of signature validity, and repeat fetches succeed. X-Amz-Expires=120 only bounds a cold cache-miss at the S3 origin.

Mitigation: don't treat 120s/single-use as a boundary; fetch server-side and never surface the link.
